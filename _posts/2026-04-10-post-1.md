# GTRDBMS - Connection and Session Model - Part 1

Since, we are starting the General Theory with a troubleshooting mindset first, we will begin with understanding how the User Connection and Session is established in general from Application to the Database Server.

Framework: Connection and Session Model
![gtrdbms-post-1-image-a](/media/images/gtrdbms-post-1-image-a.jpeg)

The connection flow goes through multiple layers.
Troubleshooting user connection issues becomes easier if you have atleast a basic understanding of each layer.

A fundamental understanding of connection flow is like this:

1. The application has a connection string which stores the address of the database server, the port number & the user credentials.
2. The DNS Resolution then tries to resolve the DB Server hostname.
3. It will be assigned an ephemeral port on the application server and attempt to establish a TCP connection to the destination. It can fail due to IP reachability issues or if no service is listening on the destination port.
4. It will also require to able to successfully pass through the Application Firewall + DB Server Firewall.
5. The TCP connection is established after successful three-way handshake, assuming no issues in SYN queue handling.
6. The TCP connection is then placed in the TCP ACCEPT queue until the database listener accepts it. It is then handed over to the database engine, where authentication and logon processing are performed.
7. Before the logon processsing & authentication, the database engine performs resource and admission control checks such as maximum connection limits, OS-level constraints (e.g., ulimit for process and file descriptors) etc. If limits are exceeded, the connection is rejected at this stage.
8. After successful authentication, a user session is created with associated session memory and assigned to a server-side execution resource (process, thread, or worker pool depending on the database architecture).
9. Once the database session exists, authorization checks are performed to validate what privileges the session has at the database level.
10. The user can now send the SQL Queries over the TCP connection that was established, the database then executes those SQL Queries and returns the Result-Set back to the user over the same established TCP Connection.

If any layer (or points) mentioned above breaks, the user connection will face or encounter issues.
Based on the user connection error, you can identify which layer might be having the issue.
