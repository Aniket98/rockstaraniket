# Oracle - Explained Filesystem Structures

I like clarity.
Extreme clarity.

Today's topic: Explaining the Oracle Filesytem Structures

What's /u01? /u02? ..... u0N?
What's /u01/app?
What's /u01/app/oracle?
What's /u01/app/oracle/product?

Why all these chains of names?

In this article, I will not talk much about what is ORACLE_BASE, ORACLE_HOME, ORACLE_SID etc - kind of commonly aware terms. I will assume you have enough understanding of atleast fundamental terms. You can even read the [Post](https://rockstaraniket.com/2026/04/11/post-2.html) to get a quick revision. In future, we can even go in detail for the basic fundamentals terms as well. But today, our attention is on something more interesting.

This article will focus on which FS, what name, what directory name, what directory tree paths to use.
Basically, how should one decide/design and create the Filesystem Structures for Oracle Products.

### Setting an Intro

Normally, we see oracle filepaths something like below:

```
/u01/app/oracle/product/19.0.0/dbhome_1
```

The above is just 1 commonly used example.
Obviously there are many other kind of directories and filepaths as well present (I am not listing them all).

Anyhow.

The question here comes : "Why am I using /u01, app, oracle etc"
Can i not simply create 1 folder like this:

```
\oracledb\dbhome_1
```

or

```
\mydb\dbhome_1
```

Who's stopping you to use a folder name like that?
The database will still work with above simplified filepath.

And the truth is : Yes, you can use this simplified filepath as well. It's fine.
Even oracle says it's not mandatory to use standard filepaths as a commonly used pattern.

This realization is important to know. It removes the fear of 'complexity'.
That this simplified filepath is allowed to use as well.

Then, why use a standard structured design?
The reason we use a "Standard Structure" is to get :

	1. Better Organization (of the Oracle directories & files!)
	2. Which helps in Easier Management/Maintenance/Administration
	3. Future Scalability (in future if i need to install more Oracle Products, or multiple Homes, or different DB versions owned by different users etc - it should be easier for me to add on or remove things on same server)

Because of better organization of things, even managing other things like Troubleshooting, Upgrades, Setting Up Multiple Homes, Database Growth, Separating software and data - become easier.

So, the very next thing that comes to mind is : "Great, I like this strategy, so how should I oraganize the things?"
And the answer is : "You follow Oracle - OFA (Optimal Flexible Architecture) guidelines."

What is OFA?

--> Oracle - OFA (Optimal Flexible Architecture) is a standard.
This standard is set of configuration guidelines to ensure well-organized Oracle installations (software + data).
This simplifies administration, support and maintenance of the oracle database product.

You can think of Optimal Flexible Architecture as a set of good habits to adopt when organizing Oracle directories and files on your computer.

OFA standard helps you to organize database software and configure databases to allow multiple databases, of different versions, owned by different users to coexist as well.

Note: This does not mean OFA standard is only meant to maintain multiple databases on a server.
Even if you have only single oracle database, still you should follow the OFA standard as it provides many benefits already described above in this article.

So, even if you have single database, Oracle recommends that you use Optimal Flexible Architecture, specially if the database will grow in size, or if you plan to have multiple databases.

Also, one of the advantages of using the OFA standard is it improves the "readability" of the oracle directories and files.

This standard provides many other benefits as well, including performance benefits (like I/O distribution), you can read more about them in the official links provided.

Here is a list of main important benefits:

1. Structured organization of directories and files, and consistent naming for database files simplify database administration.
2. Distribution of I/O across multiple disks prevents performance bottlenecks caused by multiple read or write commands issued simultaneously to a single drive.
3. Distribution of applications across multiple disks safeguards against database failures.
4. Login home directories are not at risk when database administrators add, move, or delete Oracle home directories.
5. Multiple databases, of different versions, owned by different users can coexist concurrently.
6. Software upgrades can be tested in an Oracle home in a separate directory from the Oracle home where your production database is located.

### Implementing the OFA - Optimal Flexible Architecture Standard

We will now see the naming convention strategy & other considerations that are advised in OFA Standard.

Before beginning, you must know the main locations of Oracle Database.
From a high-level OFA-style perspective, Oracle database layout is often grouped into 7 logical areas:

```
ORACLE BASE
ORACLE HOME
ORACLE INVENTORY
ORACLE ORADATA (control files, data files, redo log files)
ORACLE ADMIN --> DBA-managed(primarily human-driven) database administrative files (scripts, exports, and supporting DB files)
ORACLE DIAGNOSTICS (DIAG) --> automatic system-generated troubleshooting data (alert, traces, incidents, logs)
ORACLE RECOVERY AREA (FRA - Fast Recovery Area) --> stores and manages files related to backup and recovery.
```

Now, we will map these components onto the file system (FS) using the OFA standard.

Note: We will follow the latest OFA guidelines. Older OFA standards existed for earlier Oracle Database versions, but I am not writing them here in this article. The links to those older standards have alreaady been provided in the earlier section.

1) Naming - File Systems

***

The data will live on the file systems.
So, we start off here.

How many FS i need?
--> Oracle documenation says: "To fully implement the Optimal Flexible Architecture recommendations for a database stored on file systems that are not striped or mirrored, you require at least three file systems located on separate physical devices."

Having multiple FS is primarily to improve separation and I/O distribution.

Now, how to name the mount points for these FS.

Oracle documentation says: "Name all file system mount points using the syntax /pm, where p is a string constant and m is a unique fixed-length key (typically a two-digit number) used to distinguish each mount point. For example: /u01 and /u02, or /disk01 and /disk02."


```
# Naming FS Mount Points

/pm

p: a string constant
m: a unique fixed-length key (typically a two-digit number)

```

So, a typical exmaple of oracle files distribution on FS mount points can be:

/u01 --> oracle software & related files (ORACLE_HOME, inventory, admin base etc)
/u02 --> data files
/u03 --> redo log files or recovery files
/u04 --> optional expansion (data or redo or recovery files)
/u05 --> optional expansion as per storage planning
.. so on.

Note: These /u0N assignments are only examples. Actual file placement (oracle software, datafiles, redo files, etc) can vary and depends on your storage design, performance requirements, and DBA decisions.

Similarly, the mount points can have other string for 'p' like 'ora', which is equally valid as well:

/ora01 --> oracle software & related files (ORACLE_HOME, inventory, admin base etc)
/ora02 --> data files
/ora03 --> redo log files or recovery files
/ora04 --> optional expansion (data or redo or recovery files)
/ora05 --> optional expansion as per storage planning
.. so on.

Oracle only defines pattern here --> /pm. You can fill this pattern.
Usually, people use /u01 /u02 /u03 etc.

2) Naming - Oracle Base

***

Oracle documentation says:

"The Oracle Base directory is the top level directory that you can use to install the various Oracle software products. You can use the same Oracle base directory for more than one installation. If different operating system users install Oracle software on the same system, then each user must create a separate Oracle base directory.

Name Oracle base directories using the syntax /pm/s/u."

```
# Naming Oracle Base

/pm/s/u

pm: A mount point name
s: A standard directory name
u: The name of the owner of the directory (the user running Oracle Universal Installer)

```

For example, /u01/app/oracle is an Oracle base directory created by the oracle user.
And /u01/app/applmgr is an Oracle base directory created by the applmgr user.

Note:

Here, '/pm' is mount point name which we already discussed earlier.
's' is the standard directory name. Oracle documentation commonly uses app as the OFA-recommended value for s. Other names can be used, but that would be a deviation from strict OFA convention.
'u' is the directory owner (often called the software owner)


Oracle documentation also adds: "Placing Oracle base directories at the same level in the UNIX file system is advantageous because it enables you to refer to the collection of Oracle base directories on different mount points using a single pattern matching string, `/*/app/*`."

This means if multiple Oracle users have bases like:

```
/u01/app/oracle
/u01/app/applmgr
/u02/app/user3
/u02/app/user4
```

they all follow the same pattern:

`/*/app/*`

So, with one pattern `/*/app/*` you can refer to all Oracle Base directories at once.

Example:
```
ls -d /*/app/* could show:

/u01/app/oracle
/u01/app/applmgr
/u02/app/user3
/u02/app/user4
```

Live Example on server:
```
10:~ # ls -ld /*/app/*
drwxrwxr-x 1 oracle oinstall 98 Apr  5 15:39 /u01/app/oraInventory
drwxr-xr-x 1 oracle oinstall 86 Mar  7 15:31 /u01/app/oracle
drwxr-xr-x 1 oracle oinstall  0 Apr 18 14:04 /u02/app/user2
10:~ #
10:~ #

Note: The output also includes oraInventory because it matches the wildcard pattern `/*/app/*`, illustrating Oracle’s point that a single pattern can refer to all directories placed at that same level.
```

Yes, that’s a good summary.

OFA gives a **standard template that works across all Oracle environments and products**, mainly for consistency and predictability.

You *can* modify it to fit your needs, but then you’re trading:

* global standard readability (any DBA recognizes it instantly)
* tool/script compatibility assumptions
* consistent layout across servers

for:

* tighter ownership clarity
* custom organizational needs
* sometimes simpler local understanding

---

More precise way to think about it:

> OFA = common language for Oracle layouts
> Customization = local dialect optimized for a specific environment

---

Key idea:

* Small/simple system → customization is fine
* Large/enterprise system → standardization matters more than local optimization

---

One-line version:

> OFA is the universal standard; deviations improve local fit but reduce cross-system consistency and familiarity.
