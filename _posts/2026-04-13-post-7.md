# SQL Server - Versioning Explained

SQL Server has 3 parallel naming systems:

1. Marketing / Product name (what people say)

Example: SQL Server 2019, 2022
→ This is just branding + approximate release cycle.


2. Internal engine version (very important)

Example:

SQL Server 2019 → 15.x
SQL Server 2022 → 16.x

This is what the engine actually uses.

3. Full build number (exact patch level)

Example:

15.0.2000.5
16.0.1000.6

This includes:

Major version (15)
Minor version
Patch / CU level

These builds identify exact installed version + updates

When you run T-SQL "SELECT @@VERSION;" you will see an output like below:

``` bash
[mssql@rocky810 ~]$ sqlcmd -S localhost -U sa -No -Q 'select @@VERSION'
Password:
                                                                                                                                                                                             
------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
Microsoft SQL Server 2022 (RTM-CU24) (KB5080999) - 16.0.4245.2 (X64)
        Feb 25 2026 15:01:38
        Copyright (C) 2022 Microsoft Corporation
        Developer Edition (64-bit) on Linux (Rocky Linux 8.10 (Green Obsidian)) <X64>

(1 rows affected)
[mssql@rocky810 ~]$
```

The main line you see is:

```
Microsoft SQL Server 2022 (RTM-CU24) (KB5080999) - 16.0.4245.2
```

So, what this means?
Let's break it down, then slowly try to understand each piece and combine them together in the end.

```
Component 1: Microsoft SQL Server 2022
	|_ Marketing name (human-friendly)
Component 2: RTM-CU24
	|_ RTM → base release line
	|_ CU24 → 24th cumulative update on top of RTM
	|_ It means: “SQL Server 2022, updated up to CU24”.
Component 3: KB5080999
	|_ This is a Microsoft Knowledge Base article ID. It documents what’s inside the CU (here CU24).
	   Like it will document "Fixes, improvements, known issues, etc" in CU.
Component 4: 16.0.4245.2
	|_ 16 → major version (corresponds to SQL Server 2022)
   	   0 → minor version
	   4245 → build number
	   2 → revision number
```

Now, lets look at each component in detail:

SQL SERVER VERSIONING – COMPLETE NOTES

----------------------------------------
1. FULL VERSION STRING BREAKDOWN
----------------------------------------

Example:
Microsoft SQL Server 2019 (RTM) - 15.0.2000.5

Break it into parts:

1. SQL Server 2019
   → Marketing / product name
   → Human-friendly label (based on release cycle, not exact versioning logic)

2. RTM
   → Release type (servicing stage)

3. 15.0.2000.5
   → Actual engine version (most important for DBAs)


----------------------------------------
2. VERSION NUMBER STRUCTURE
----------------------------------------

Format:
Major.Minor.Build.Revision

Example:
15    . 0     . 2000    . 5
|       |        |         |
Major   Minor    Build     Revision

----------------------------------------
3. EACH COMPONENT EXPLAINED
----------------------------------------

(A) MAJOR VERSION (15)
- Represents SQL Server generation
- Mapping:
  15 → SQL Server 2019
  16 → SQL Server 2022

- Used internally by engine and tools

---

(B) MINOR VERSION (0)
- Usually stays 0 for main releases
- Rarely used in SQL Server versioning

---

(C) BUILD NUMBER (2000)  ← MOST IMPORTANT

Definition:
A build is a compiled, packaged version of the software.

Why called "build"?
- Software is "built" (compiled) from source code
- Each compilation snapshot = a build

Purpose:
- Identifies a specific released version
- Used for:
  - Patch tracking
  - Support cases
  - Upgrade decisions

Examples:
- 2000 → RTM
- 4000+ → CU builds

Key point:
- Every public update → NEW BUILD number

---

(D) REVISION NUMBER (.5)

Definition:
A revision is a small internal iteration of the same build.

Why called "revision"?
- Means a small modification or rework of the same version

Purpose:
- Tracks internal recompilations of same build
- Used by Microsoft engineers for precision

Example:
15.0.2000.1
15.0.2000.2
15.0.2000.5 ← final released version

All are:
- SAME build (2000)
- Different revisions

Key point:
- Only ONE revision is usually publicly released
- Others are internal/testing

----------------------------------------
4. BUILD vs REVISION (CORE DIFFERENCE)
----------------------------------------

Build:
- Public-facing
- Changes with every release (RTM, CU, etc.)
- What DBAs should track

Revision:
- Internal detail
- Rarely matters in real-world DBA work

---

Simple analogy:

Build = Version (v1, v2, v3)
Revision = Internal tweak (v1.0.1, v1.0.2)

---

Practical rule:

- Compare BUILD numbers in production
- Ignore REVISION unless debugging edge cases

----------------------------------------
5. RTM, CU, SP, GDR (SERVICING MODEL)
----------------------------------------

(A) RTM (Release to Manufacturing)
- First official release
- Base version

Example:
15.0.2000.x → SQL Server 2019 RTM

---

(B) CU (Cumulative Update)

Definition:
A package that includes ALL previous fixes.

Meaning of "Cumulative":
Each CU includes:
- Its own fixes
- All previous CU fixes

Example:
CU1 → Fix A
CU2 → Fix A + B
CU3 → Fix A + B + C

So:
You install ONLY latest CU

---

(C) SP (Service Pack)
- Older model (pre-2017 mainly)
- Large bundled updates
- Mostly replaced by CU model

---

(D) GDR (General Distribution Release)

Definition:
Security-only update track

Contains:
- Critical security fixes ONLY

Does NOT include:
- Performance fixes
- General bug fixes

---

----------------------------------------
6. PATCHING PATHS
----------------------------------------

Two main paths:

1. CU Path
   - Frequent updates
   - Includes:
     - Security fixes
     - Bug fixes
     - Performance improvements

2. GDR Path
   - Minimal updates
   - Includes:
     - Security fixes only

Important:
- CU path is most commonly used
- GDR is for highly stable/regulated environments

---

----------------------------------------
7. IMPORTANT BEHAVIOR RULES
----------------------------------------

1. Microsoft does NOT usually release multiple revisions of same build
   Example (NOT typical):
   - 15.0.2000.5
   - 15.0.2000.8

2. Instead:
   New fixes → NEW BUILD number

   Example:
   - RTM → 15.0.2000.5
   - CU1 → 15.0.4003.x

---

3. Revision numbers are mostly internal
   - You typically see only one revision publicly

---

4. Always compare BUILD numbers in production

---

----------------------------------------
8. PRACTICAL DBA TAKEAWAYS
----------------------------------------

- Always track:
  → Major version
  → Build number

- Ignore:
  → Minor version (mostly)
  → Revision (almost always)

- When troubleshooting:
  → Compare build numbers with Microsoft docs

---

----------------------------------------
9. QUICK SUMMARY
----------------------------------------

Build:
= Released version of SQL Server
= Changes with every update

Revision:
= Internal iteration of same build
= Mostly irrelevant for DBAs

CU:
= Cumulative (includes all previous fixes)

GDR:
= Security-only updates

RTM:
= First release version

---

----------------------------------------
10. SELF-CHECK QUESTIONS (FROM YOUR DOUBTS)
----------------------------------------

Q1: Why 2000.5 (two numbers)?
→ 2000 = build, 5 = revision

Q2: Will Microsoft release multiple revisions of same build?
→ Almost never publicly

Q3: If fix comes, revision or build changes?
→ Build changes

Q4: What should I compare in production?
→ Build number

Q5: If on RTM and latest is CU10?
→ Install CU10 only (because cumulative)

Q6: If on GDR and need performance fix?
→ Not possible via GDR → need CU path

----------------------------------------
END

SQL Server has 3 version identities:

Marketing name (what you hear)
Internal version (like 16.x)
Release year

Core mapping (most important):

SQL Server 2008 → 10.0
SQL Server 2008 R2 → 10.5
SQL Server 2012 → 11.0
SQL Server 2014 → 12.0
SQL Server 2016 → 13.0
SQL Server 2017 → 14.0
SQL Server 2019 → 15.0
SQL Server 2022 → 16.0

What “16.x” really means:

16 = major version (SQL Server 2022)
x = minor updates (CU, patches)
e.g. 16.0.1000 → base release
16.0.4100 → later cumulative update

Important pattern (this is what DBAs remember):

Every new release increments major version by +1
After 2017 → releases became more frequent (no long gaps like before)

Gotcha (interview favorite):
SQL Server 2008 R2 = 10.5, not 11
→ only time Microsoft used a “.5” major version
