# SQL Server - Versioning Explained

When you run T-SQL "SELECT @@VERSION;" you will see an output like below:

``` bash
[mssql@rocky810 ~]$ sqlcmd -S localhost -U sa -No -Q 'select @@VERSION'
Password:
                                                                                                                                                                                             
------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
Microsoft SQL Server 2022 (RTM-CU24) (KB5080999) - 16.0.4245.2 (X64)
        Feb 25 2026 15:01:38
        Copyright (C) 2022 Microsoft Corporation
        Developer Edition (64-bit) on Linux (Rocky Linux 8.10 (Green Obsidian)) <X64>

(1 rows affected)
[mssql@rocky810 ~]$
```

The main line you see is:

```
Microsoft SQL Server 2022 (RTM-CU24) (KB5080999) - 16.0.4245.2
```

So, what this means?
Let's break it down first, then slowly try to understand each piece and combine them together in the end.

```
Component 1: Microsoft SQL Server 2022
	|_ Marketing name (human-friendly) - The year in which the First Major Version was launched.
Component 2: RTM-CU24
	|_ RTM → base release line
	|_ CU24 → 24th cumulative update on top of RTM
	|_ It means: “SQL Server 2022, updated up to CU24”.
Component 3: KB5080999
	|_ This is a Microsoft Knowledge Base article ID (KB article). It documents what’s inside the CU (here CU24).
	   It will document "Fixes, improvements, known issues, etc" in CU.
Component 4: 16.0.4245.2
	|_ 16 → major version (corresponds to SQL Server 2022)
   	   0 → minor version
	   4245 → build number
	   2 → revision number
```

Now, lets look at each component in detail:

----------------------------------------
Component 1. - Marketing/Product Name
----------------------------------------

Example: SQL Server 2019, 2022
→ This is just branding + release cycle.

If you take a look at names, you will get this:

SQL Server 2000
SQL Server 2005
SQL Server 2008
SQL Server 2012
SQL Server 2014
SQL Server 2016
SQL Server 2017
SQL Server 2019
SQL Server 2022
SQL Server 2025

A question comes here: Where is year 2024, 2023? And 2021, 2020? etc so on.
When a New Major Version gets released in a particular year, it receives that year name.

So, it means in year 2024, 2023, 2021, 2020, 2018 .. so on, a NEW Major Version was never released.
That is why you see the gap in years.

But SQL Server does 1 amazing thing more, they keep another track of Major Version which is sequential in nature.
You can call this 'Internal Engine Version'.

| Market/Product Name  | Major Version |
|:---------------------|:--------------|
| SQL Server 2000      |:    8.0       |
| SQL Server 2005      |:    9.0       |
| SQL Server 2008      |:    10.x      |
| SQL Server 2008 R2   |:    10.5      | <----- EXCEPTION
| SQL Server 2012      |:    11.0      |
| SQL Server 2014      |:    12.0      |
| SQL Server 2016      |:    13.0      |
| SQL Server 2017      |:    14.0      |
| SQL Server 2019      |:    15.0      |
| SQL Server 2022      |:    16.0      |
| SQL Server 2025      |:    17.0      |


This gives you double benefits.
You get the YEAR in Product Name.
And get the Sequential Major Version Number as well.

You can reference anyone per as your convenience.
So good.

Note : For the above exception, we will talk at end of article.

----------------------------------------
Component 2. - Update Level
----------------------------------------

```
Microsoft SQL Server 2022 (RTM-CU24) (KB5080999) - 16.0.4245.2 
```

In this, the 2nd component is "(RTM-CU24)".

RTM stands for 'Release to Manufacturing'.
It is simply the First Base Version, when the Major Version was firstly launched.

Over the time, enhancements & hotfixes might be needed (which includes updates/fixes on security, supportability, manageability, and reliability).
Those are applied through Updates called as "Cumulative Update (CU)".

What does "Cumulative" mean?
--> If you open dictionary, the meaning of this word is : "including all the amounts that have been added before".

To put it simply, lets say a New Major Version is launched.
The first launch is called RTM (Base Version).

Over time, new security patches/bug fixes keep coming like this:

CU 1
CU 2
CU 3
CU 4
CU 5
CU 6
CU 7
CU 8

Every CU brings more fixes.
Now, let's say you are having (RTM-CU4). This means after 1st Base release, 4 CUs have been applied.

After CU 4, if you directly apply CU 8, it will automatically include the fixes introduced in CU-5, CU-6 and CU-7.
This is called "CUMULATIVE". 

Means CU 8 contains all fixes of previous CUs as well.
It also gives benefit you can directly apply the latest CU, and it will include all previous CUs fixes as well.

More Simple Definition of CU:
--> A package that includes ALL previous fixes.

Each CU includes:
- Its own fixes
- All previous CU fixes

Example:
CU1 → Fix A
CU2 → Fix A + B
CU3 → Fix A + B + C

This clears the (RTM-CUx).

Now, there can be some organizations that want only security fixes without any other changes.
Out of CU, they only want security fixes, and no other improvements introduced in CU.

Due to this a new service path was created called "GDR" - General Distribution Release.
It only contains security fixes.

Hence, the latest servicing model of SQL Server divides into 2 paths:

		_____ CU
		|
RTM ----
		|
		----- GDR

You can choose the path as per your need.

So, to write simply, there are two main SERVICING/PATCHING paths:

1. CU Path
   -- Frequent updates
   -- Includes:
     -- Security fixes
     -- Bug fixes
     -- Performance improvements

2. GDR Path
   -- Minimal updates
   -- Includes:
     -- Security fixes only

Important:
-- CU path is most commonly used
-- GDR is for highly stable/regulated environments

Component 3. - KB Article
----------------------------------------

```
Microsoft SQL Server 2022 (RTM-CU24) (KB5080999) - 16.0.4245.2 
```

The 3rd component is (KB5080999).

A SQL Server KB (Knowledge Base) article is an official Microsoft support document (identified by a "KB" number, e.g., KB123456) that details specific software updates, bug fixes, security patches, or troubleshooting information.

Every new CU comes with new KB article number.

You can see the official list of KBs here: [Link](https://support.microsoft.com/en-us/topic/kb957826-where-to-find-information-about-the-latest-sql-server-builds-43994ba5-9aed-2323-ea7c-d29fe9c4fbe8?utm_source=chatgpt.com)

----------------------------------------
Component 4. - Internal Enging Versioning + Build & Release
----------------------------------------

```
Microsoft SQL Server 2022 (RTM-CU24) (KB5080999) - 16.0.4245.2 
```

The 4th component is: "16.0.4245.2"

--> Major.Minor.Build.Revision

Example:
16    . 0     . 4245    . 2
|       |        |         |
Major   Minor    Build     Build Revision


Official [Link](https://learn.microsoft.com/hi-in/sql//sql-server/install/work-with-multiple-versions-and-instances-of-sql-server?view=sql-server-ver17#version-components-and-numbering) says:

The standard product version format for SQL Server is MM.nn.bbbb.rr, where each segment is defined as:

MM - Major version
nn - Minor version
bbbb - Build number
rr - Build revision number

##### Major Version

Major version is the primary SQL Server release number that identifies a specific product generation (for example, 16.0 for SQL Server 2022).

We have alread discussed this in component 1.
This is the Internal Major Version Number, which is used internally by sql server engine and tools.

##### Minor Version

-- Usually stays 0 for main releases
-- Rarely used in SQL Server versioning

To introduce fixes/enhancements, instead of incrementing minor version, SQL Server relies generally on CU or GDR.
That's why you will almost always see Minor Version only "0" (like 12.0, 13.0, 14.0 etc.)

##### Build Number

Definition:
A build is a compiled, packaged version of the software.

Why called "build"?
-- Software is built (compiled and packaged) from source code into a deployable binary.
-- Each unique compiled output + packaging state is called a build.

To say it more simply, a BUILD is a version of "software produced" after compiling and packaging the source code into a deployable release.

Note: Build Number is not CU.
==> Build number is the version "IDENTIFIER" assigned to a specific compiled release of SQL Server after a particular update (RTM or CU) is built and packaged.

Example: The first base release (called RTM) without any CU will also have a build number.
Like "16.0.1000.x" --> Here 1000 is build number that came with RTM release.

On top of this, if any CU/GDR is applied, then this build number can increment like:

		RTM		---------> 16.0.1000.x
		RTM-CU1 ---------> 16.0.2304.x
		RTM-CU2 ---------> 16.0.2653.x
		RTM-CU1 ---------> 16.0.2993.x

What is the Purpose of Build Number:
-- Identifies a specific released version
-- Used for:
  -- Patch tracking
  -- Support cases
  -- Upgrade decisions

##### Build Revision Number

Definition:
A build revision is a small internal iteration of the same build.

Why called "revision"?
- Means a small modification or rework of the "same" build.

Does the Build Revision Number matter for you a DBA?
|_ Before publishing the BUILD, the Microsoft Developers/Engineers work on developing the BUILD by compiling code. They might need to do repeated internal testings and fixes for the same build, which are tracked with Revision Numbers.

Once a build is fully tested and stabilized, it is published with a specific "build number + revision number".

As a DBA, you do not work with internal revision iterations.
Revision numbers matter more to the Microsoft Internal Engineers.
As a DBA, you would have main focus on the Build Number. So, you generally look at it collectively "Build + Revision".


BUILD vs REVISION (CORE DIFFERENCE)

Build:
-- Public-facing
-- Changes with every release (RTM, CU, etc.)
-- What DBAs should track

Build Revision:
-- Internal detail
-- Rarely matters in real-world DBA work


----------------------------------------
PRACTICAL DBA TAKEAWAYS
----------------------------------------

-- Always track:
  → Major version
  → Build number

-- Ignore:
  → Minor version (mostly)
  → Revision (almost always)

-- When troubleshooting:
  → Compare build numbers with Microsoft docs

----------------------------------------
SELF-CHECK QUESTIONS
----------------------------------------

Q1: Why 2000.5 (two numbers)?
→ 2000 = build, 5 = revision

Q2: Will Microsoft release multiple revisions of same build?
→ Almost never publicly

Q3: If fix comes, revision or build changes?
→ Build changes

Q4: What should I compare in production?
→ Major version and Build number

Q5: If on RTM and latest is CU10?
→ Install CU10 only (because cumulative)

Q6: If on GDR and need performance fix/enhancement updates?
→ Not possible via GDR → need CU path
