# Linux - Number of Process and File Descriptors Limit

#### ------------------------- No of tasks (processes + threads) 

--- check process limit of shell
su - user
ulimit -u

--- change process limit of ONLY YOURS shell
ulimit -u 9000
Here, 9000 is new value.

--- check process limit of a RUNNING process
cat /proc/<pid>/limits

--- increase/decrese soft:hard limit of a RUNNING PID
prlimit --pid 3734 --nproc=15:20

Here, 15 is new soft limit and 20 is new hard limit.

!!! WARNING !!!: Changing soft:hard limit for a RUNNING PID can be risky, so try to avoid during runtime.

!!! NOTE !!!: Changing soft:hard limit for a RUNNING PID will not be inherited again for already forked child processes. Only the new child forked processes will inherit new limit. You will need to set the new process limit of already existing child forked processes manually using prlimit (if needed).

--- change GLOBAL ulimit for a specific user (this is good adoption for DBAs to set database environent before starting database)

Add below lines to /etc/security/limits.conf :-
postgres   soft   nproc   Add this:
10000
postgres   hard   nproc   15000

This will ensure every new postgres shell will get these new limits.

Also, check directory: /etc/security/limits.d/

--- Very Important: Setting NPROC Limit in systemd
Show current value (if set explicitly):
|_ systemctl show postgresql | grep LimitNPROC

Else, set by editing service file:
|_ Add this:
[Service]
LimitNPROC=15000

Then:-
systemctl daemon-reload
systemctl restart postgresql

--- TRY TO AVOID! :- Setting shell level ulimit (less reliable for services and does not apply to non-interactive shells)
In ~/.bashrc  or  ~/.bash_profile, set:-
ulimit -u 10000

--- Additional Note:

ulimit -u impacts process-based connection models directly
(Postgres, Oracle dedicated)

Thread-based systems (MySQL, SQL Server) are mostly unaffected for connections,
but still limited indirectly at extreme OS limits

Postgres/Oracle → limited by memory + processes
MySQL/SQL Server → limited by memory + threads

But in the Linux kernel, processes and threads are treated similarly as "tasks" or "runnable entities".
Both processes + threads are "work units" for LINUX.
So, nproc ulimit will affect both numbers of processes + threads created by a user.

--- Hence, you should better use below command to count os tasks (processes + threads)
ps -L -u <username> | wc -l

Note: -L stands for lightweight process (LWP).
In Linux terminology, a "lightweight process" is essentially another name for a thread.

Note: You might also see processes in output as a standalone process with only one thread (look at LWP column).

--- To see thread names as well
ps -u <username> -L -o pid,lwp,comm

#### ------------------------- Open File Descriptors

--- check the maximum number of open file descriptors (files & sockets) that a single process (or the current shell) is allowed to have at once
su - user
ulimit -n
cat /proc/<pid>/limits

Note: The other settings should be similar as above shown for ulimit -u (work units limit)
