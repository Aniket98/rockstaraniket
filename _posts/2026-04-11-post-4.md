------------------------------ Ephemeral ports ----------------------------

Ephemeral ports
|_ “Ephemeral” = temporary / short-lived

In networking:
|_ ephemeral port = temporary port used by client side
	|_ Example:
			Client → uses random port (e.g. 49152)
			Server → listens on fixed port (e.g. 5432)

Ephemeral port exhaustion
|_ Decides how many outbound connections can a machine create.

--- see ephemeral ports limit
cat /proc/sys/net/ipv4/ip_local_port_range
|_ Output example: 

``` bash
[root@localhost ~]# cat /proc/sys/net/ipv4/ip_local_port_range
32768   60999
[root@localhost ~]#
```

This means --> 32768–60999 → ~28K ports
So, ~28K concurrent outbound connections max.
If exhausted, new connections cannot be created.

---------------------------- TCP Handshake ----------------------------------------

A TCP handshake is how a client and server establish a reliable connection before sending data. Think of it as a polite “hello” where both sides agree:

Client wants to talk
Server agrees
Client confirms

After this, data can flow reliably.

Step-by-step with packets (3-way handshake)

1. SYN (synchronize)
	Client → Server
	“Hey, I want to start a connection, here’s my initial sequence number”
2. SYN-ACK (synchronize-acknowledge)
	Server → Client
	“Got your request, here’s my sequence number, and I acknowledge yours”
3. ACK (acknowledge)
	Client → Server
	“Got your reply, connection is now established”

------------------------------ TCP Backlog - SYN Queue (Half-Open Connections Queue) ----------------------------

SYN = Synchronize

Before full connection, connections sit in SYN queue.
SYN queue exists on server side.

It stores half-open connections (handshake not finished yet).

--- SYN queue limit controlled by:
cat /proc/sys/net/ipv4/tcp_max_syn_backlog
|_ example:

``` bash
[root@localhost ~]# cat /proc/sys/net/ipv4/tcp_max_syn_backlog
128
[root@localhost ~]#
```

If this queue fulls: → handshake fails → client retries/timeouts

------------------------------ TCP Backlog - Accept Queue ----------------------------

Client → SYN → Server → SYN-ACK → ACK → connection ready (handshake completed)

After handshake, connections wait in accept queue until DB picks them.

--- accept queue limit controlled by:
cat /proc/sys/net/core/somaxconn
|_ example:
``` bash
[root@localhost ~]# cat /proc/sys/net/core/somaxconn
2048
[root@localhost ~]#
```

------------------------------ TIME_WAIT ----------------------------

After closing connection, socket stays in: TIME_WAIT (few seconds/minutes)

Why socket stays? → ensure delayed packets don’t corrupt new connections
Problem: You open & close lots of connections quickly, then ports stay “occupied” → leads to ephemeral exhaustion.

Which side TIME_WAIT exist? Client or Server Side?
|_ Who closes the connection? The side that actively closes enters TIME_WAIT.

In DB cases:
|_ usually client closes first → so client side goes into TIME_WAIT.

--- check connections existing in TIME_WAIT
``` bash
ss -ant | grep TIME-WAIT
```

--- check configured value of TIME_WAIT timeout
cat /proc/sys/net/ipv4/tcp_fin_timeout
|_ example: 
``` bash
		[root@localhost ~]# cat /proc/sys/net/ipv4/tcp_fin_timeout
		60
		[root@localhost ~]#
```

This is often used as a reference, though TIME_WAIT is technically based on TCP rules (~2×MSL).
[MSL = Maximum Segment Lifetime]
